# 마케팅 성과 인사이트 리포트 자동화 파이프라인

> **"숫자를 요약한 리포트"가 아니라, 마케팅팀·경영진이 받은 자리에서 '이번 주 예산을 어디로 옮기고 무엇을 손대지 말지'를 결정하게 만드는 주간 의사결정 도구를, 새 CSV만 갈아 끼우면 동일 품질로 재생성되는 파이프라인으로 만들었다.**

- Basic → Standard → Challenge 누적 채점 기준(총 160점) 기준 설계·구현
- 계산은 전부 Python(`src/`), Claude는 검증된 수치의 해석만 담당
- 마케팅팀 상세본 / 경영진 요약본 2종을 **하나의 계산 정본**에서 렌더 → 문서 간 수치 불일치 구조적으로 0
- 새 CSV 투입 시 정제 → 계산 → 이슈 탐지 → 리포트 생성이 자동 재현, 4종 변형 데이터로 자동 검증

## 1. 문제 정의 — 이 과제를 어떻게 재해석했는가

과제 원문은 "마케팅 성과 인사이트 리포트 생성"이지만, 실제로 풀어야 할 문제는 다음과 같이 재정의했다.

> 숫자를 예쁘게 요약하는 게 아니라, **리포트를 받은 사람이 그 자리에서 '이번 주 예산을 어디로 옮기고, 무엇을 손대지 말아야 하는지'를 결정하게 만드는 것** — 리포트와 의사결정 사이의 간극을 없애는 것.

### 왜 이렇게 봤는가 — 이 데이터가 스스로 증명하는 함정 3가지

주어진 `data/marketing_performance.csv`를 naive하게 "총매출·ROI 표"로만 요약하면, 실제로 세 번 잘못된 결정을 유도한다.

| # | 함정 | 이 데이터에서 실제로 벌어지는 일 | 대응 설계 |
|---|------|-------------------------------|-----------|
| 1 | **액션 부재** | 총매출 4.4억·MER 862%·전환 22,858건을 봐도 "그래서 이번 주에 뭘 하라는 건가"가 없다 → 리포트를 받고 다시 회의를 열어야 함 | '이번 주 업무 우선순위' 표 신설 (§4) |
| 2 | **ROI 절대순위 오판** | 이메일 채널이 ROI 1025.98%로 절대 1위 → "증액하자"가 자연스러운 결론이지만, 시장 벤치마크(ROAS) 대비로는 **개선여지** 등급 → 절대 순위만 보면 잘못된 채널에 증액 | 채널 평가를 내부 순위가 아닌 **시장 벤치마크 대비**로 전환 |
| 3 | **손실과 데이터 왜곡의 혼동** | 이메일 매출이상치 2,884만원을 '손실'로 읽으면 엉뚱한 예산 삭감으로 이어지지만, 실제로는 원본 재확인이 필요한 **데이터 왜곡**(예산 손실 아님) | 이슈를 '예산 레버' vs '데이터·트래킹 레버'로 조치처 자체를 분리 (§6, Step 43) |

방치하면 리포트는 "읽고 넘기는 문서"가 되고, 예산은 근거 없이 배분되며, 진짜 회수 가능한 기회손실(메타광고 과집행 882만원)은 방치된다.

**성공 기준 재정의.** 이 프로젝트에서 '잘함'의 기준을 **"수치가 많다"가 아니라 "받은 사람이 추가 회의 없이 이번 주 액션을 결정할 수 있다"**로 바꿨다. 이 기준이 아래 모든 설계 결정을 지배한다.

## 2. 설계 구조 — 북극성 목표에서 산출물까지

전체 설계를 지배하는 단일 기준(북극성 목표, [docs/REPORT_DESIGN.md](docs/REPORT_DESIGN.md) 참조):

> **리포트를 받는 대상(마케팅팀·경영진)이, 이 문서 하나만으로 자신이 가장 궁금했던 점과 필요했던 정보를 추가 작업·추가 질문 없이 즉시 얻고, 곧바로 자기 업무를 실행할 수 있어야 한다.**

이 목표에서 두 가지 구조적 결정이 도출된다.

**(1) 독자가 둘이다 → 산출물을 2종으로 분리한다 (계산 1벌, 렌더 2벌)**

| 산출물 | 독자 | 담는 것 | 뺀 것 |
|--------|------|---------|-------|
| [exec_report](output/exec_report.md) | 경영진 (승인) | 한눈에 보기 · 최근 2주 모멘텀 · 예산·매출 직결 이슈만 · 예산 재배분 **결정 요약** | 데이터 정정·품질 이슈(노이즈) |
| [insight_report](output/insight_report.md) | 마케팅팀 (집행) | 채널 진단(ROI·CPA·CTR·CVR) · 주차별 ROAS 추이 · 전체 이슈 + 조치 · 벤치마크 · 예산 재배분 **상세 기획안** | — |

두 문서 모두 `reallocate.run_pipeline()`이라는 **동일한 계산 정본**을 렌더만 다르게 소비하므로, 경영진 보고서와 마케팅팀 보고서의 숫자가 어긋나는 일이 구조적으로 불가능하다.

**(2) 섹션 순서 = 거시 → 미시, 현상 → 조치**

```
핵심 요약 → 이번 주 업무 우선순위 → 데이터 개요 → 채널 효율 평가 → 추세 → 이슈 → 예산 재배분 기획안 → 의사결정 로그
```

변화 비교는 **좁게(전주 대비) → 넓게(8주 추이)**로 줌아웃해 단일 주 비교의 노이즈를 8주 흐름으로 보정한다. 상세 근거는 [docs/REPORT_DESIGN.md](docs/REPORT_DESIGN.md) §1·§2에 지표별로 정리했다(왜 ROI를 3종으로 쪼갰는지, 왜 이슈를 원(₩) 임팩트로 정렬하는지, 왜 CPA를 추가했는지 등).

## 3. 논리 구조 — 파이프라인 아키텍처

```
data/marketing_performance.csv
        │
        ▼
┌───────────────────────────────────────────────────────┐
│ reallocate.run_pipeline()  — 정본 계산 1벌               │
│   ├─ calculate.py     합계·ROI·CTR·CVR·변화율·주차별 ROAS │
│   │                   최근 8주 롤링 윈도우 트림           │
│   ├─ detect_issues.py LOO 중앙값 기준선 + 적응형 임계값   │
│   │                   으로 급등/급락/이상치/결측 자동 탐지 │
│   └─ 예산 재원·재배분안 산출 (회수 가능액 → 우선순위 매칭) │
└───────────────────────────────────────────────────────┘
        │  (동일한 dict 결과를 두 렌더러가 공유)
        ├───────────────────────────┬─────────────────────
        ▼                           ▼
 generate_report.py           generate_html.py
 (markdown 렌더)                (정적 html 렌더)
        │                           │
        ▼                           ▼
 insight_report.md            insight_report.html
 exec_report.md                exec_report.html
        │
        ▼
 tests/verify_reproducibility.py
 (원본 + 변형 4종에서 자동 검증)
```

- **판정과 감시를 축으로 분리**했다: 이상치·이슈 탐지는 강건한 *다른 주차 중앙값* 기준(LOO, leave-one-out), 급변 모니터링은 *전주 대비(WoW)*. WoW만으로 이상치를 판정하면 전주 노이즈에 왜곡되므로 판정에는 쓰지 않는다.
- **계산과 해석을 레이어로 분리**했다: `calculate.py`/`detect_issues.py`가 숫자와 1차 분류를 확정하면, `generate_*.py`는 그 결과를 문장·표로 옮길 뿐 새로운 계산을 하지 않는다.

## 4. 핵심 산출물 — "이 표만 보면 이번 주 업무 분배가 끝난다"

리포트를 '숫자 요약'에서 그치지 않고 바로 손을 움직일 수 있는 실행 문서로 설계했다. 그 증거가 마케팅팀 상세본의 **이번 주 업무 우선순위** 표다 — '핵심 이슈' 섹션의 근거를 순위·담당·시점으로 한곳에 취합했다.

| 순위 | 할 일 | 근거 | 담당 | 시점 |
|------|-------|------|------|------|
| 1 | 메타광고 과집행 차단 후 회수분을 네이버광고에 재배분 | 초과지출 3,194,281원, 순증 상한 12,772,685원 | 마케팅 리드 | 즉시 |
| 2 | 이메일 매출이상치 원본 재확인·보정 | 왜곡·추정 28,838,361원 (예산 손실 아님) | 트래킹팀 | 이번 주 |
| 3 | 네이버광고 revenue결측 원본 재확인·보정 | 왜곡·추정 1,385,548원 (예산 손실 아님) | 트래킹팀 | 이번 주 |

(전체 6개 조치는 [insight_report.md §이번 주 업무 우선순위](output/insight_report.md) 참조)

**예산 재배분 기획안**은 근거 나열에서 그치지 않고 **문제 정의 → 재원(회수 가능 손실) → 재배분안 → 우선순위 판단 기준**까지 포함해 마케터가 추가 회의 없이 착수할 수 있게 설계했다. 우선순위는 새 스코어를 만들지 않고 채널 ROI·이슈 임팩트·시장 벤치마크 등급 등 **이미 계산된 값과 근거만** 사용한다. 경영진에게는 이 기획안의 결정 요약(승인 요청)만 [exec_report.md](output/exec_report.md)로 전달한다.

**이슈 섹션**은 5차례 설계를 거쳐(§6 Step 42→43) 최종적으로 **조치처(레버) 단일 기준 3그룹**으로 정리했다 — 예산(재배분 대상) / 운영·크리에이티브(채널 담당 확인) / 데이터·트래킹(원본·수집 확인, 예산 조치 아님) + 참고용 긍정 신호. 각 그룹 안에서는 매출 임팩트(₩) 큰 순으로 정렬해 "무엇부터 조치할지"가 바로 드러난다.

## 5. 코드 구현

```
src/
├── calculate.py       합계·ROI·CTR·CVR·CPA·전주 대비 변화율·주차별 ROAS 계산
├── detect_issues.py   급등/급락/이상치/결측/기회 신호를 규칙 기반으로 탐지·분류(레버 태깅)
├── reallocate.py       예산 재원 산출(회수 가능액) + 우선순위 매칭 재배분안 생성
├── generate_report.py  위 결과를 마케팅팀 상세본/경영진 요약본 markdown으로 렌더
└── generate_html.py    같은 결과를 정적 HTML 대시보드 2종으로 렌더(md와 로직 동일, 표현만 다름)

tests/
├── make_fixtures.py            원본 CSV에서 4종 변형(A~D) 생성
└── verify_reproducibility.py   원본+변형 4종에 대해 파이프라인 전체를 실행, 필수 구성요소 존재 여부 자동 검증
```

**이상치 탐지 — modified z-score.** 평균·표준편차 기반 z-score는 이상치 자체가 평균을 끌어당겨 둔감해지므로, 중앙값·MAD(median absolute deviation) 기반 modified z-score(threshold 3.5)를 사용해 소수 데이터에서도 강건하게 탐지한다.

**급등/급락 탐지 — LOO(leave-one-out) 중앙값 대비 배수.** "이번 주가 유독 큰가"를 판정할 때 이번 주 자신을 포함한 평균/중앙값과 비교하면 이상치 자신이 기준선을 오염시킨다. 따라서 판정 대상 주차를 제외한 나머지 주차의 중앙값을 기준선으로 쓰고, 그 대비 배수가 채널·지표별 적응형 임계값을 넘는지로 급등/급락을 판정한다.

**레버 태깅 (`_lever()`).** 이슈 유형 문자열을 조치 담당(레버)으로 매핑하는 단일 분기 함수 — `'결측'/'이상치' → 데이터·트래킹`, `'광고비' → 예산`, `'성과' → 운영·크리에이티브`, 그 외 → 전략. 이 한 함수가 이슈 섹션 전체의 그룹핑 기준이 된다(§6 Step 43에서 2축 분류를 이 1축으로 수렴).

**md/html 동형성 유지.** 콘텐츠 로직(무엇을 보여줄지, 어떤 순서로 정렬할지)은 `generate_report.py`에서 계산해 `generate_html.py`가 그대로 import(`from generate_report import ...`)해 재사용한다 — 표현(마크다운 문법 vs HTML 태그)만 각 파일에서 따로 담당해, 두 렌더러가 서로 다른 결론에 도달할 여지를 없앤다.

## 6. 진행 과정 — 43번의 설계 결정

모든 판단 근거는 [decisions.md](decisions.md)에 배경/결정/변경 파일/검증 형식으로 빠짐없이 기록했다. 방향을 바꾼 주요 결정만 추리면:

| Step | 결정 | 왜 |
|------|------|-----|
| 초기 | 손실 이슈와 데이터 정정 사안 분리 | 이상치·결측을 모두 '손실'로 묶으면 조치처가 다른데도 같은 취급을 받는다 |
| 초기 | 채널 평가를 내부 순위 → 시장 벤치마크 대비로 전환 | 절대 ROI 1위여도 시장 대비로는 개선여지일 수 있어 증액 오판 위험 |
| 초기 | 결측을 0으로 채우지 않고 NaN 유지, 집계에서 제외 | 0으로 채우면 평균·ROI가 왜곡됨 |
| 초기 | 독자별 2종 리포트로 분리 (계산 1벌, 렌더 2벌) | 경영진(승인)과 마케팅팀(집행)의 필요 고도가 다름 |
| Step 38→39 | 목차·섹션 번호 추가 후 완전 제거 | "이게 독자의 다음 행동을 앞당기는가?" 기준으로 재검토 → 인덱싱 자체가 목적이 아니라고 판단, 유사 섹션 병합(9개→6개)만 남김 |
| Step 40→41 | '지금 당장 할 일' 체크리스트를 플레이북에 흡수했다가 완전 삭제 | 플레이북 표의 할 일·근거 컬럼과 완전히 중복 — 같은 내용을 두 형태로 반복하지 않는다는 원칙 적용 |
| Step 42 | 이슈 카드에 레버 태그 + 긴급도 기반 정렬 추가 | "분류 기준이 불명확하다"는 피드백에 1차 대응 |
| **Step 43** | 이슈 분류를 5그룹(2축: 카테고리×레버) → **3그룹(레버 1축)**으로 재조정 | Step 42로도 "읽는 사람 입장에서 중구난방"이라는 피드백 → 재점검 결과 '운영 관찰'과 '데이터 품질'이 사실상 같은 레버 분리를 두 번 반복 적용한 것이었음을 확인, 조치처 하나의 축으로 수렴하고 임팩트(₩) 내림차순 정렬로 통일 |

이 과정에서 드러난 협업 방식의 특징: **분류 기준을 처음부터 완벽하게 설계하지 않고, "표준인가?"라는 스스로의 질문에 정직하게 "아니오, 커스텀 기준이다"라고 답한 뒤 그 답에서 드러난 약점(2축 중복)을 근거로 재설계**했다 — 이 판단 과정 자체를 decisions.md에 남겨, 최종 구조뿐 아니라 그 구조에 도달한 이유도 추적 가능하게 했다.

## 7. 내가 특별하게 가져간 것 (강점)

1. **문제를 그대로 풀지 않고 재정의했다.** "인사이트 리포트를 만들어라"는 과제를 그대로 수행하면 수치 요약 문서가 나온다. 이 데이터셋 자체가 유도하는 3가지 오판(§1)을 먼저 증명한 뒤, 그 오판을 막는 것을 성공 기준으로 재정의하고 모든 섹션 설계를 그 기준에 맞춰 역산했다.
2. **계산 정본 하나로 독자별 2종 문서를 만들어 "수치 불일치"를 설계로 원천 차단했다.** 별도 파일에서 각각 계산하지 않고 `run_pipeline()` 하나의 결과를 md/html, 경영진/마케팅팀 4개 산출물이 공유한다.
3. **설계를 한 번에 확정하지 않고, 사용자 피드백을 근거로 3번 리팩터링했다(Step 38→39, 40→41, 42→43).** 특히 Step 43은 "이 분류가 업계 표준인가?"라는 질문에 방어적으로 답하지 않고 실제로는 커스텀 기준이며 구조적 중복이 있다는 것을 스스로 인정한 뒤 재설계로 이어갔다 — 이 판단 과정 전부가 [decisions.md](decisions.md)에 남아있다.
4. **판정(강건)과 감시(민감)를 의도적으로 다른 통계 기준으로 분리했다.** 이상치/이슈 탐지는 LOO 중앙값(전주 노이즈에 흔들리지 않음), 모멘텀 모니터링은 전주 대비(WoW, 변화를 빠르게 포착). 하나의 지표로 두 목적을 억지로 겸하지 않았다.
5. **재현성을 "된다"가 아니라 "자동 검증됨"으로 증명했다.** 원본 CSV 외에 4종 변형(짧은 기간·소채널, 오가닉 없음, 주차 재라벨링, 8주 초과 12주 트림)을 만들어 `verify_reproducibility.py`로 매번 회귀 검증한다 — 새 CSV가 들어와도 하드코딩된 가정이 깨지지 않는다는 근거다.
6. **Challenge 요구사항(예산 재배분안)을 별도 부록이 아니라 마케팅팀 리포트 안에 문제정의-재원-재배분안-우선순위 판단 기준까지 완결된 하나의 의사결정 문서로 흡수했다.**

## 9. 실행 방법

```bash
# 마케팅팀 상세본 + 경영진 요약본 (md 2종)
python3 src/generate_report.py [입력CSV] [출력MD]

# 위 둘의 정적 HTML 대시보드 2종
python3 src/generate_html.py [입력CSV]

# 재현성 자동 검증 (원본 + 변형 4종)
python3 tests/verify_reproducibility.py
```

인자를 생략하면 `data/marketing_performance.csv`를 읽어 `output/`에 생성한다. 새 CSV를 넣으면 최근 8주 분석 창·변화율 비교 주차·오가닉 섹션 유무·재배분안 개수가 데이터에 맞게 자동으로 조정된다(하드코딩 없음).

## 10. 제약 · 의도된 스코프

과제 제약을 그대로 준수했다.

- **계산은 전부 Python** — 합계·ROI·변화율·재배분을 수기 계산에 위임하지 않는다. Claude는 완성된 수치의 해석만 담당한다.
- **실시간 API 연동·예측 모델·웹앱 제외** — 제공된 `marketing_performance.csv`만 사용, 서버·Streamlit 없이 정적 md/html만 산출한다.
- **이미지 차트 제외** — HTML은 자바스크립트·서버 없이 파일 하나로 열리는 정적 문서이며 md 리포트가 정본이다.
- **새 리드 스코어 미신설** — 우선순위 판단은 기존 계산값·근거만 사용한다.
